§1上下文模型:同一个键,不同场景不同含义

调研里最值得抄的不是任何一张键位表,而是 Notion / VS Code 共同的作用域模型:好的编辑器不发明一堆冷僻组合键,而是让一小撮键在明确的「当前场景」里复用。场景由两件事决定:焦点在哪(编辑器 / 侧栏 / 弹层)和编辑器处于哪个态(光标打字中 / 块被选中)。

弹层(最优先)modal / 菜单 / 命令面板开着时,↑↓ Enter Esc 全归它,其余键不下发Esc 永远=关掉它
编辑器 · 文本态光标在块里闪:格式键、markdown 触发、斜杠菜单、Enter/Tab/Backspace 的块行为Esc → 降到块态
编辑器 · 块选中态块作为整体被选中(无光标):方向键选块、Delete 删块、Cmd+D 复制块、Enter 回文本态Esc → 取消选中
侧栏 / 文件树树内 ↑↓ 移动、Enter 打开、重命名/删除(后续)焦点在树上才生效
应用壳(全局)标签页、保存、快速打开、侧栏开关、缩放——任何焦点都响应
系统层(碰不得)Cmd+Q/Tab/Space/H/M、输入法组合态,以及 §4 的系统文本键

三条派发原则(实现时的硬规则,也是 review 时判断「这个键行为对不对」的依据):

  1. 最内层命中即消费:同一个 Cmd+D,块选中时=复制块;弹层开着时不响应;没人认领时必须放行给浏览器/系统。没匹配到就不 preventDefault——这是 §4 系统键存活的前提。
  2. Esc 是逐层退出的阶梯,永不做破坏性动作:关弹层 → 文本态退到块态 → 取消块选中 → 什么都不做。Notion / VS Code / Bear 全是这个设计。
  3. Enter 是逐层进入:块选中态按 Enter 进入文本态;树里按 Enter 打开文件;和 Esc 互为反向。
Notion 的双态模型(我们的块编辑器已经隐性拥有):ui-demo 的 Canvas 已经区分 editingId(文本态)和 selectedId(块选中态),Esc 降档、方向键跨块也已实现——本设计不是引入新概念,而是把这个已有模型的键位补全、并把规则写明。

§2五个裁决点(需要拍板)

调研发现的真冲突:两个阵营的肌肉记忆抢同一个键,必须选边。每条给出建议,不同意的话改这里,§3 总表会跟着变。

裁决 1 · Cmd+1…9 给「标签直达」还是「标题」?

写作 app 阵营(Bear/Typora/iA)把 Cmd+1…6 给标题;浏览器阵营(Chrome/Safari)给标签直达。Notion 两边都不占:转块走 Cmd+Option+数字。

建议:跟 Notion——转块用 Cmd+Option+0…6,把 Cmd+1…9 留给标签直达(Wordspace 有浏览器式标签页,正好各得其所)。

裁决 2 · Cmd+E:行内代码 vs 导出 PDF(ui-demo 与真 app 已打架)

Notion 的 Cmd+E=行内代码,ui-demo 也已这么实现;但真 app 已把 Cmd+E ship 给了「导出 PDF」(v0.5.0 菜单加速键)。同一个键在两个产品面里含义不同,迟早撞车。

建议:编辑场景赢——Cmd+E=行内代码(Notion 肌肉记忆),导出 PDF 挪到 Cmd+Shift+E。真 app 需要跟着改一行菜单加速键。

裁决 3 · Cmd+F:文档内查找 vs 文件筛选(真 app 现状)

所有文档编辑器(Word/Docs/Notion)的 Cmd+F=当前文档内查找;真 app 目前把它绑成「聚焦侧栏文件筛选框」(因为还没有文档内查找功能)。跨文件/全局搜索的业界惯例是 Cmd+Shift+F(VS Code/Obsidian/Bear/iA 全一致)。

建议:把 Cmd+F 的名分留给「文档内查找」(功能后续做);文件筛选迁到 Cmd+Shift+F。趁现在没人有肌肉记忆,早迁早好。

裁决 4 · 删除线:Cmd+Shift+X vs Cmd+Shift+S

Word(mac) 和 Google Docs 一致用 Cmd+Shift+X;Notion 独走 Cmd+Shift+S,而 macOS HIG 里 Cmd+Shift+S 的正统含义是「另存为/Duplicate」。ui-demo 目前跟的 Notion。

建议:删除线改 Cmd+Shift+X(两票对一票,X 也更像「划掉」),把 Cmd+Shift+S 还给「另存为」(真 app 的另存为目前只在 ⋯ 菜单里、无快捷键,正好补上)。

裁决 5 · Cmd+K 与命令面板归属(基本无悬念,确认一下)

五家写作 app(Bear/Obsidian/iA/Craft/Typora)全票 Cmd+K=插入链接,这是写作类铁律;web 产品(Slack/Linear)才拿它做 omnibox。真 app 已把 Cmd+P 给了快速打开(不是打印),方向正确。

建议:维持 Cmd+K=插入链接、Cmd+P=快速打开文件;将来若做「命令面板」(带动作搜索),用 Cmd+Shift+P(VS Code 双键流),Cmd+P 不动。打印/导出走 Cmd+Shift+E。

§4系统白送的文本键:不许破坏清单

这些键来自 macOS 文本系统,在任何 contenteditable 里天生就有,用户肌肉记忆最深。我们一行代码都不用写,但任何全局 keydown 拦截都可能砸掉它们——判定标准:用户在任意输入框按 Cmd+Backspace,若触发了自定义功能而不是「删到行首」,用户会认为 app 坏了。

键含义
Cmd+← / Cmd+→,Cmd+↑ / Cmd+↓行首/行尾,文档首/尾(+Shift=选区扩展)
Option+← / Option+→按词跳(+Shift=按词选)
Option+Backspace / Cmd+Backspace删一个词 / 删到行首
Ctrl+A/E/K 等 Emacs 系行首/行尾/删到行尾(Cocoa 标配)——别把 Ctrl+A/E/K 挪作他用
Cmd+C/X/V/AElectron 陷阱:这几个靠应用菜单的 Edit role 路由,自定义菜单时必须保留,否则编辑器里复制粘贴直接失灵(真 app 已正确保留)
输入法组合态(拼音候选中)Enter/方向键在 isComposing 时必须放行给输入法——中文产品的生死线(块编辑器已守,新增键一律要带此守卫)

§6ui-demo 本轮落地清单(= Wendi 验收清单)

Colin 对 §2 裁决点无异议后,以下逐条在 ui-demo 实现;Wendi 验收时照此清单逐条打勾。

新增键位(全部已实现 ✅)

  1. 标签页:Ctrl+Tab/Ctrl+Shift+Tab 循环、Cmd+1…9 直达、Cmd+W 关标签(浏览器抢占则 Electron 生效)
  2. 格式:删除线迁 Cmd+Shift+X、Cmd+Shift+V 纯文本粘贴
  3. 块操作:Cmd+D 复制块、Cmd+Shift+↑/↓ 移动块、Cmd+Option+0…6 转块、Cmd+Enter 待办打勾

边界修复(调研时发现的既有问题,已一并修掉 ✅)

  1. Cmd+Z 劫持:焦点在侧栏输入框(筛选/改名/地址栏)时,编辑器不得抢撤销——按 §1 原则加焦点作用域守卫
  2. SaveModal 的 Enter:焦点在「取消」按钮上按 Enter 不应触发保存——Enter 只在默认焦点链上确认
  3. 全局键的弹层守卫:modal 开着时 Cmd+S/Cmd+P 等全局键不应穿透执行
  4. 新增键全部带输入法守卫(isComposing)与「未命中放行」原则,§4 系统键逐条复测不被破坏

明确不做 + 理由(防 review 时误判为遗漏)

  • Cmd+N:「新建」统一走 Cmd+T 标签页心智,不设第二入口(真 app 同)
  • Cmd+Y 当重做:Windows 遗产,mac 上 Chrome 拿它开历史,统一 Cmd+Shift+Z
