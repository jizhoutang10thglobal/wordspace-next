# Wordspace × Chromium：说人话的调研（给 Colin）

写给非技术读者。回答三个问题：①我们用的是不是 Chromium？②Chrome 的功能能不能都搬过来？③浏览器和文档怎么才能不割裂、变成一个东西。

---

## 一句话结论

- **内核就是 Chromium，而且整个 app 都跑在它上面**——文档和网页用的是同一个渲染引擎，这是「能合成一体」的技术地基。
- **Chrome 的功能不能全搬**：引擎（画页面、跑 JS、PDF、缩放、打印这些）几乎白拿；但 Chrome 那层「谷歌云服务」（防钓鱼、账号同步、扩展商店、翻译、密码管理）不在引擎里，搬不过来。
- **文档和浏览器不该硬合成一个渲染机制**（技术上会更脆），**该做的是「让接缝看不见」+ 加一座桥：把网页一键存成本地文档**。这才是让它变「一个工具」的关键，而且不贵。

---

## 一、我们用的是 Chromium 吗？是。而且是整个 app。

Electron 这个框架 = **Chromium（谷歌浏览器的引擎）+ Node（后端能力）** 打包在一起。关键点：

- 你打开一个**本地文档**，是 Chromium 把 HTML 画出来的。
- 你打开一个**网页**，也是 Chromium 把 HTML 画出来的。
- **同一个引擎**。所以「文档和网页本来就是一回事（都是 HTML）」这句话，技术上是成立的地基。

你现在觉得「两张皮」，是因为我目前用了**两套装法**：文档装在一个受限的「iframe 沙箱」里（安全、只读脚本、能编辑），网页装在一个独立的「Chromium 视图」里（能跑任意网站）。两套叠在一起，才显得割裂。**这不是做错了**——见第三节，业界（VS Code、Electron 官方）都是这么分的，分开是对的；要改的是「让用户感觉不到接缝」。

---

## 二、Chrome 的功能能全搬吗？——三个桶

把 Chrome 的能力分三类。判断标准：**引擎自带的白拿，云服务搬不了。**

### 🟢 白拿（引擎自带，几行代码或本来就有）

前进/后退/刷新/历史、缩放（还自带「同域名网页共享缩放」）、打印 + 导出 PDF、**内置 PDF 阅读器**（Chromium 自带 PDFium，跟 Chrome 一样）、开发者工具、拼写检查、截图、Cookie/缓存清理、**沙箱 + 站点隔离**（Chrome 的核心安全架构，我们自动就有）、无障碍、画中画、媒体控制。

→ 这些要么已经有，要么接一下线就有。

### 🟡 要做（引擎干重活，我们做界面）

页内查找（引擎搜，我们做那个查找条）、**下载**（引擎下载，我们做下载进度条 + 管理）、全页截图、**阅读模式**（Chromium 的没开放，但业界成熟做法=注入 Mozilla 的 Readability 库，Min 浏览器就这么干）、**少量扩展**（能装解包的扩展、部分 `chrome.*` 接口能用，靠社区库 `electron-chrome-extensions`；但不是「随便装商店里任何扩展」）。

→ 这些是正常浏览器的产品工作量，按需做。下载/查找/阅读模式通常是前三个该做的。

### 🔴 不值得 / 做不到（谷歌的云服务层，不在引擎里）

- **⚠ 防钓鱼/防恶意网站（Safe Browsing）——没有，这是最重要的安全缺口。** Chrome 会弹「此网站有欺诈风险」，**我们不会**，因为这是谷歌的云端黑名单服务，Electron 根本没打包。**如果让用户随便输网址上网，他们的安全防护是低于 Chrome 的。** 应对：要么接谷歌的 Safe Browsing API 自己查，要么限制只能开信任的站，最起码得跟用户说清楚「这不是加固过的浏览器」。**这条必须让你知道。**
- **翻译**（谷歌的翻译是云服务，引擎里没有——要翻译得接第三方 API 如 DeepL）
- **自动填充 + 密码管理**（Chrome 那套连的是谷歌账号，搬不过来；要做得自己做加密密码库，或靠系统/1Password）
- **跨设备同步**（书签/历史/密码同步 = 谷歌服务器 + 谷歌账号，做不了，除非我们自建后端）
- **谷歌账号登录、Chrome 扩展商店、Chrome 自己的自动更新系统**（都是谷歌专有；自动更新我们已经用 electron-updater 自己做了，签名公证那套）

### 给你的一句话

> **「用 Chromium」= 拿到整个浏览器引擎（世界级、几乎白拿），但拿不到 Chrome 上面那层谷歌云服务（防钓鱼、同步、商店、翻译、密码管理）——那些是谷歌服务器上的东西，从来不在盒子里。**
> 所以「照搬 Chrome 全部功能」真实的意思是：**引擎全搬（能），谷歌的云重建（不能）。**

业界的 Electron 浏览器（Min、Wavebox）都是这么取舍的：引擎白拿的全用，云服务不硬追，需要就接第三方或自己做个小的。真需要完整 Chrome 生态的（Wavebox）会离开 Electron 直接基于 Chromium 源码——这本身就说明 Electron 给不了「完整的 Chrome」。

## 三、怎么让浏览器和文档「不割裂、变成一个东西」

这是你最在意的。调研了 Arc、Notion、Beaker、Obsidian、VS Code 的做法，结论分两层。

### 架构层：**别硬合成一个渲染机制，业界都是分两套的**

- **VS Code**：它自己的界面直接渲染（信任），但插件给的任意 HTML 一律关进 iframe 沙箱 + 强制 CSP + 只能消息通信——**故意分两个信任等级**，防止不可信内容逃逸。跟我们「本地文档可信可编辑 vs 远程网页不可信要沙箱」是同一个问题，同一个解法。
- **Electron 官方**同样建议：按信任等级用不同装法（自己人的内容用 iframe，第三方网站用独立视图）。
- **结论**：「文档和网页用同一套机制」是一句**北极星式的比喻**（都是 HTML），不是要真的把渲染引擎合成一个。硬合会更脆、更不安全。**当前分两套是对的工程。**

那「变成一个东西」靠什么？——**靠让接缝看不见** + **加一座桥**。

### 交互层：三招让它变「一个工具」（按性价比排序）

**1. 🌉 把网页「存成本地文档 / 剪藏进工作区」——这是最关键的一座桥（强烈建议做）**

这是让「浏览」从一个**并列功能**变成**文档库的进料口**的关键动作。参考：
- **Arc 的 Capture**：在网页上框选一块内容，直接拖进你的笔记/画布，还保持可交互——不是截图。纯界面层，不用新引擎。
- **Notion / Craft / Obsidian 的 Web Clipper**：浏览时把当前页直接存进你的文档树（选个文件夹或已有文档当落点），转成可编辑的内容而不是死链接。

对 Wordspace：**「当前网页 → 存成一个本地 HTML 文档（走我们的 Schema 校验分流）」**。用户浏览到有用的东西，一键变成自己文档库里能编辑的文档。这一下子就把浏览器和文档焊在一起了——浏览是为了喂养你的文档。**这是整份调研里性价比最高的一招。**

**2. 🔤 一个输入框，两件事（我们已经做了一半）**

Arc 的做法是一个 Cmd+T 命令栏，同时搜标签/历史/书签/直接导航——一个输入框让两个子系统「感觉像一个工具」。我们现在有两个入口：侧栏 omnibox（输网址上网）+ Cmd+P 命令面板（搜文件）。**建议合并方向**：让 Cmd+P 也能输网址/搜索词（除了搜文件），或让 omnibox 也能搜本地文件。一个框干两件事，割裂感立刻少一半。

**3. 🎨 共享外壳：标签、主题、导航钮完全一套**

Arc 是「最激进的融合者」，但它也没把网页和笔记塞进一个列表——笔记在「Library」、标签在侧栏，是**相邻但轻分隔**。它之所以还像一个产品，是因为**导航栏、主题、侧栏 chrome 完全共享**。我们已经在做这个方向（网页标签和文档标签在同一个侧栏标签栏、同一个 omnibox、同一套主题）。继续保证：网页态和文档态的顶栏/侧栏/交互**长得一模一样、行为一致**，切换时只有内容变、框架不变。

### 一个反面教材

Notion 是「万物皆块」的代表，但它**也没有**把「实时网页」和「原生文档」合成一个渲染路径——网页嵌入永远是个带框的只读盒子，跟文档块视觉上分开。**连 Notion 都做不到真·统一渲染**——这告诉我们「一个机制」这条别钻牛角尖，把力气花在「接缝看不见 + 存成文档的桥」上。

Beaker 浏览器（2017-2022，已停）是跟我们**最像**的前身：一个「编辑即浏览」的浏览器。它 UX 理念没失败，是**一个人扛不动整套 P2P 后端**才停的——**这是范围教训，不是方向教训**：方向对，别把摊子铺太大。

---

## 建议的下一步（优先级）

1. **先跟你确认安全口径**：内置浏览器没有 Safe Browsing。v1 是「限信任站 / 加提示 / 接第三方 API」哪种？（默认我先加个「这不是加固浏览器」的提示 + 外部协议已封死）
2. **做「网页 → 存成本地文档」这座桥**（最高性价比，让它真变一个工具）。
3. 继续保证网页态/文档态**共享外壳、接缝看不见**（本轮已在修割裂 bug）。
4. Cmd+P 与 omnibox 逐步合并成「一个框两件事」。
5. 白拿的能力（缩放/打印/PDF 阅读器/查找/下载/阅读模式）按需接线。
