## 模板为什么够格（rationale）

The template earns "pipeline-grade" by fusing the EXISTING board trio (which already exists at projectx-board/pm/templates/) with the demo's machine-checkable VA, then closing three leaks the raw fusion still has.

PRECISION: every §5.2 Given-When-Then [P1] line maps to >=1 F##-<slug>.va.json check with a falsifiable threshold on a REAL metric — computed-style (bgLuminance fail-closed when alpha<0.99 so dead-CSS can't read as "very dark"; bgColor exact rgb equals), textContent (contains/notContains/equals), or geometric getBoundingClientRect (projectx already ships toolbar-no-overlap.spec.ts). Thresholds pin the SPECIFIC numbers from §3 既定约束 (F15: 400% ceiling, 25% floor, pointer-anchor invariant, per-doc isolation), never "zoom works" — because shipped useZoom (0.5-2.0 stepped global top-center) would pass a "works" gate, the exact S4 weak-assertion trap. The strength test: if you can imagine a CSS-dead/wrong-but-shipped app that still passes, the assertion is too weak. CRITICAL precision FIX the raw fusion lacks: va-coverage today only checks the .va.json EXISTS, so a spec author can satisfy the gate with cheap already-shipped checks and silently omit the load-bearing [P1] lines (catastrophic for F15). So the template requires each [P1] line to carry an explicit check-id tag (e.g. [P1][va:f15-ceiling]) and extends va-coverage to assert every tag resolves to a check — converting R2 from convention to a vitest-red mechanism.

JUDGE!=PLAYER: three mechanical locks, not convention. (1) The intent card (F##-<slug>.intent.md, human-authored/frozen BEFORE code) is the external yardstick the gate measures success against (board B4) — the implementer cannot redefine success. (2) The implementer AI authors ZERO assertions: it writes feature code + the Electron/DOM-decoupled pure-logic module only; the spec-agnostic va-runner reads all strength from the human va.json. (3) dev/.github/CODEOWNERS (ABSENT today in both projectx and projectx/dev) locks va.json/va-runner/va-selftest/va-eval/va-coverage/gate.yml to Colin. But the strongest guarantee is TEMPORAL, proven by the demo's own commits (4954c27 "VA 输入（人写，实现前置）"): the pipeline must HARD-FAIL if the va.json (and any new va-eval metric it needs) is not committed by Colin BEFORE the implementer branch exists. CODEOWNERS-approve-after-the-fact is strictly weaker than write-before — an /lfg implementer blocked by a red coverage check has every incentive to draft a metric+tolerance ("I added pointer-anchor with ±3px") that a rubber-stamp review converts player->judge. So new metrics are forbidden on the implementer branch; a CI step diffs va-eval.js against the pre-implementation baseline and auto-rejects (not just review-requests) any implementer change.

UNATTENDED-RUNNABILITY: §5 names the REAL harness with zero deferrals — fast gate `npm test` (vitest run) in projectx/dev; `npm run build:all` before any real-Electron VA (config hard-fails if dist-server/server.cjs missing — stale build runs stale server); browser e2e `npm run test:e2e`; CI gate.yml. The template declares the VA run target PER spec (R7): pure-renderer specs (F14/F15/F46) use the Vite-baseURL browser path where projectx's strong gates already run; only preload/IPC features use _electron.launch. It mandates a fixture-open step in the va.json schema because the real app has NO content at page-load (the demo's runner snapshots immediately — that's why the port is a rewrite). It requires an In-Scope testable logic layer decoupled from Electron (a module not importing electron) so the headless container vitest gate has something to bite. And EVERY "由 dev 决定 / 由设计稿决定" PRODUCT/UX deferral is eliminated — nailed into §3 or pushed to §2 Out — because an unattended agent has no downstream human (the board ruleset's B3 grep enforces this; un-gated F06 carries 5 such deferrals, gated F15/F40 are clean — that's the bar). Impl-detail deferral ("具体 API/库由 dev 决定") stays allowed. The mutation self-test (va-selftest) is MANDATORY: strip all CSS + empty validated text, assert the VA goes RED, else the whole e2e job fails — making "is the assertion strong enough" a tested object, the one thing projectx wholly lacks.
