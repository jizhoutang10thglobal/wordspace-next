# 非合规 HTML 文件的基础编辑

## 1. 问题框定

Wordspace 打开一个「非合规 / 野生 HTML」文件（含 `<script>` / 绝对定位 / 合并单元格 / 块上内联 `style` / `h5`-`h6` / `iframe` 等，确定性校验器 `schemaCheck` 判为不符合 Schema #1）时，走「降级为基础编辑」这条路。

现状（`ui-demo/src/components/BasicEditor.tsx`）：文件在沙箱 iframe 里全保真渲染，`body` 挂 `contentEditable`，选中文字用 `execCommand` 做 加粗/斜体/下划线/删除线。除此之外没有任何能力——UX 太裸、也没有「哪能编辑 / 哪只读」的感知。

目标：**即便文件不合规，也尽力让用户把「能编辑的部分」编辑好，且在不破坏原文件样式 / 排版 / 定位的前提下**；同时把编辑 UX 做好看。核心难点是 Colin 点出的那个问题——**要为「编辑操作」确立一套确定性的后端语义，并适配到所有非合规 HTML**。

## 2. 角色与范围

- **先落地在 ui-demo**（React 原型，UX 参考标准）：改造 `BasicEditor`。真 app（`src/**`）后续照 ui-demo 跟进。
- 用户 = 打开一份野生 `.html` 想改点东西的人（改错字、改颜色、删掉一段），但不想（也不该）把它的排版搞乱。
- 立场沿用校验器脊梁：**磁盘字节 reparse 出的 DOM 是唯一真相**，不跑文档 JS，不信文件自称。

## 3. 编辑能力（冻结）

三个能力，其余明确不做：

### A. 富就地文字编辑（保留 + 加料）
- 点击一段真实文字 → 就地 `contentEditable` 编辑；结构 / 样式 / 定位一律不动。
- 选中文字浮出工具条：**加粗 / 斜体 / 下划线 / 删除线 + 文字色 + 高亮 + 清除格式**。
  - 明确**不加**：链接、行内代码（Colin 定）。
- 上下方向键在「文字编辑中」= 浏览器原生光标（跟着视觉行走）。

### B. 删除整块（从元素操作里只留这一个）
- 选中一个「块元素」→ 只能**删除整块**。
- 明确**砍掉**：拖拽重排、复制 / 复刻（对非合规是多余的；重排不该在降级路径硬做）。

### C. 空间切块（方向键在块之间移动焦点）
- 不在文字编辑态时，方向键 `↑ ↓ ← →` 在「块」之间移动**焦点块**。
- **切块逻辑 = 空间导航（按视觉方向找最近的块）**，不是 DOM 顺序：
  - 用渲染后的屏幕矩形（`getBoundingClientRect`）算每个块中心，筛出按键方向那一侧的，挑「方向距离 + 侧向偏移惩罚」最小的。
  - 因此：左右分栏 `→` / `←` 对；绝对定位角标按它在屏幕上的真实位置可达；**天然不会从顶跳到底**。
  - **前端不标序号**，用户不需要知道块的顺序。
  - 方向模糊的边角情形（一个块四周都有邻居、按某个方向该去谁）**不追求完美**——挑一个合理的、别乱跳就行。
- 焦点块上：`Delete`/`Backspace` = 删除（走能力 B）；`Enter` = 进入它的文字编辑（走能力 A）。

**最终形态 = A + B + C 三者合一**：点字改字（A）→ 需要时退出编辑用方向键 / 手柄在块间走（C）→ 焦点块删除（B）。参考交互见原型 `edge-lab.html` 的「A+B 组合」模式。

## 4. 后端操作语义模型（Colin 点名要确立的核心）

**文档模型** = 文件 reparse 出的 DOM 树（source of truth = 磁盘字节）。**每个用户操作 = 一次锁定到节点身份的局部 DOM 变更**；保存 = 把 DOM 重新序列化回 `.html`，其余字节尽量原样不动。「一个操作对应一套后端逻辑、适配所有非合规 HTML」——就是下面这几条：

| 操作 | 后端语义 | 键 / 触发 |
|---|---|---|
| 改文字 | 就地改某个文字节点的内容 / 包一个行内标记（`<strong>`/`<em>`/`<u>`/`<s>`/带 class 的颜色·高亮 span）；只动这个节点 | 点击 → 编辑 |
| 加行内格式 | 在选区上加 / 去行内标记，keyed 到选区所在节点 | 选中 → 工具条 |
| 删整块 | 从 DOM 移除该块节点及其子树 | 焦点块 `Delete` / 工具条 🗑 |
| 切换焦点块 | `next = nearestInDirection(当前块, 方向)`，方向距离基于 `getBoundingClientRect` 的渲染矩形（**不依赖 DOM 顺序**） | `↑↓←→`（非编辑态） |

关键点：**导航 / 定位用「渲染后的视觉几何」，编辑 / 删除用「节点身份」**。二者都不依赖脆弱的 DOM 顺序——这正是让「一套逻辑适配所有非合规 HTML（含 z-index / 绝对定位 / 左右布局）」成立的原因。

**只读（改不了的东西）**：图片等非文字元素；CSS 生成的内容（`::before`/`::after`、背景图里的字）——「看得见但不是真文字节点」的一律只读，并给出可感知的 🔒 提示。

## 5. 各 edge case 的边界（验收锚点）

以原型 `edge-lab.html` 的 9 个 edge case 为准，最终形态（A+B+C）逐条：

| edge case | 期望行为 |
|---|---|
| 普通文字 | ✅ 点字改 + 富格式 |
| 含链接 / 加粗的混排段 | ✅ 改字，链接和 `<b>` 保留 |
| 带内联 style 的提示框 | ✅ 改里面的字，背景色 / 边框内联样式保住 |
| 左右分栏（文件序≠视觉） | ✅ 文字直接改；`←→` 在两栏间走（视觉对） |
| 合并单元格表格 | ✅ 单元格改字；整表可删；不硬拆合并格 |
| 图片 + 叠放说明（z-index） | ✅ 叠字能改；🔒 图片只读；整块可删 |
| CSS `::before` 生成的字 | 🔒 CSS 字改不了；后面真文字能改；整行可删 |
| 三层嵌套段落 | ✅ 点最内层改字；焦点 / 删除作用在最外层块 |
| 绝对定位角标（文件排最后） | ✅ 改字；方向键按它的屏幕真实位置到达；可删 |

## 6. 明确不做（Non-goals）

- ❌ 拖拽重排、复制 / 复刻块。
- ❌ 在前端标块的顺序序号。
- ❌ 按 DOM 顺序做方向键切换（会乱跳、且 `←→` 一维对不上）。
- ❌ 工具条里加链接、行内代码。
- ❌ 把非合规文件「修」成合规 / 结构化（那是合规块编辑器的事，走 `feat/schema-1`）。
- ❌ 跑文档 JS、加载外链样式。

## 7. UX 方向

- 保留「全保真沙箱 iframe 渲染」——打开像打开任何 `.html`。
- 顶部一条轻降级提示（不展开违规清单）。
- 悬停高亮「可编辑」区；只读元素给 🔒 感知。
- 富文字工具条复用编辑器同款浮条样式（`.ws-fmtbar`）。
- 空间切块的焦点用清晰的焦点框；不打断、不弹序号。

## 8. 成功标准 / 验收

1. 打开原型里的每个 edge case，第 5 节逐条行为都成立。
2. 改字 / 删块后，未触及的部分**样式 / 排版 / 定位不变**（重新序列化不破坏其余字节）。
3. 空间切块：左右分栏 `←→` 正确、绝对定位块按屏幕位置可达、无「从顶跳到底」。
4. 只读元素（图片 / CSS 生成字）不可编辑且有提示。
5. `npm run build` 通过；ui-demo 里能上手跑通全流程。

## 9. 开放问题（留给实现 / 后续）

- A+B 里「点字=编辑、方向键=切块」的切换手势：靠 `Esc` 退出编辑后方向键才切块；是否要更顺（如退出编辑后自动把焦点落在该块）——实现时打磨。
- 空间导航的侧向惩罚权重（现原型 `cross*2`）在真实文档上的手感微调。
- 真 app（`src/**`）落地时，编辑作用在沙箱 iframe 的 document 上、保存时如何最小 diff 回写文件——ui-demo 是 mock，真 app 需要真读写；本轮先不解决。
- 表格 / 嵌套的「块粒度」在个别文档上可能仍有歧义，按第 3 节 C 的「不追求完美、别乱跳」处理。
