## 2. 慢模式（提速机会，按安全度分级）

### 2a. waitForTimeout 硬睡 —— 全仓 194 个（app.spec 独占 78）
硬睡是「等固定毫秒」而非「等某条件」,既慢又脆(慢机器上偶发 flake)。**但不是全可删**:
- **可安全替换**:等一个「有可观测终态」的东西(元素出现/类切换/文本变化)→ 换 `waitForSelector`/`expect.poll`,更快更稳、**零断言改动**。
- **载荷型、暂留**:等一个「无可观测信号」的东西(1.2s 自动保存 debounce 落盘、动画无终态标记)→ 没有更好的信号前保留,别硬删(删了变 flake)。
分布:app 78 / doc-links 14 / images 13 / browser 11 / paged 9 / align 8 / sticky-ancestor 7 / appearance 7 / …
**建议**:分批小 PR,每次一个 spec,逐个判「这条 waitForTimeout 在等什么」,能换的换、载荷的留;每 PR 该 spec 改前后全绿 + 抽一条变异自检。app.spec 的 78 个是最大单块,单独一轮。

### 2b. 每测重启 → beforeAll 合并 —— 高收益但高风险，**不自动做**
app 61 / tabs ~54 / sidebar ~48 / browser 35 次运行时重启是主耗时。理论上合并成单启动能省很多,**但**:
- **无安全先例**:真·beforeAll 的只有 images/paged/update-ui 小孤立 spec;大状态 spec 全故意保隔离。
- **新假绿类**:共享实例下,前一条测试留的状态(建了根 R)会让后一条即使 add-root 退化成 no-op 也假绿。
- **可行性实测**:tabs 有 10/44 是重启持久化测试(结构上不可合)、sidebar 41 条全改共享 seed 工作区——**这两个基本不可转**;app.spec 相对干净但仍 59 条状态测试。
**建议**:**默认不转**。真要转某 spec,该 PR 必须附「**乱序独立性证明**」=乱序跑==隔离跑结果一致(单点变异自检证明不了跨测污染),且逐条 Colin 拍。收益远小于 61/45/41 的启动数暗示的。

## 4. 弱断言候选（S4:查 class ≠ 视觉验证；建议升级为 computed-style/geom，非删除）

粗扫「只 toHaveClass、无 computed/geom」的 spec(可能有弱代理):`window-close-and-reveal`(6/0)、`workspace`(1/0)、`multi-root`(6/0)、`cold-start`(1/0)。**注意**:很多 toHaveClass 是**合法状态检查**(如 `.is-active` 标签态,本就是行为断言不是视觉),不是弱代理。**要逐条看**:能想出「CSS/视觉全废但断言还过」的情形才算弱。建议核这几个 spec 的 toHaveClass 是不是在替代本该读 computed-style 的视觉断言,是则升级(照 immersive/sidebar-typography 的强断言样板)。

## 5. 建议的执行批次（每批一个小 PR，都带验证）

1. **B1（安全,先做）**:app.spec 的 waitForTimeout 逐条判、能换的换 waitForSelector/expect.poll。最大单块提速,零断言改动,风险低。
2. **B2（安全）**:其余 spec 的 waitForTimeout 分批(doc-links/images/browser/paged…)。
3. **B3（人工拍板）**:弱代理断言升级(§4)——每条给「CSS 全废断言仍过」的反例证明它弱,再升级或让位给强断言。
4. **B4（人工拍板 + 乱序证明）**:beforeAll 合并——仅对能过乱序独立性证明的 spec,收益有限,谨慎。
5. **B5（人工拍板)**:冗余簇(§3)逐簇核,真冗余(变异双红+幸存≥候选强)才删。
