# Spec(draft·待排期):Split View 两栏分屏

> 状态:**draft,等 Colin/Wendi 拍板排期**。出处 = Arc UX 调研判定表「该抄(高优)」唯一大件——Arc 全部功能里几乎零差评的一个(拖拽+分屏是旧心智,不收学习税);Colin 2026-07-08 已认可方向。前置:feat/browser-tabs(PR #132)合并。**建议独立排期,别和小活混**——是这批里工程量最大的。

## 一句话

把侧栏的标签/文件拖进编辑区左/右边缘 → 吸附成左右两栏("左资料右文稿");中间一条 hairline 分隔线可拖宽窄;Cmd+W 关焦点格回单栏。

## 为什么(产品理由)

- 「边看资料边写」是文档编辑器的第一场景,浏览器标签合流后"左网页右文档"是融合的杀手用法。
- Arc 实证:唯一全员叫好的大件,因为完全骑旧心智(macOS 分屏/拖拽)。
- 现状:两个文档对照只能开两个窗口(没有多窗口!)或来回切标签。

## UX 细节

- **进入**:从侧栏拖标签行/文件行进编辑区,靠近左/右 30% 边缘 → accent 半透明吸附高亮(现有拖拽插入线同款语言),松手成分屏。**不做**菜单/快捷键进入(v1 只拖拽,保持单一心智)。
- **分隔线**:1px hairline(纸方墨圆:一条细线,不做花哨手柄),hover 变 accent、可拖 30%-70%,双击回 50/50。
- **焦点格**:点哪格哪格是焦点(细 accent 顶边指示);Cmd+S/Cmd+W/Cmd+F/面包屑都作用于焦点格。Cmd+W 关焦点格 → 回单栏。
- **组合**:文档×文档 / 文档×网页 / 网页×网页 都允许;同一文档不允许开两格(v1,避免双向编辑冲突)。
- **侧栏行为**:分屏中的两个标签都显示激活态变体(实底 + 边框区分焦点格)。
- **v2 再说**:四格、钉住整组布局、拖标签在格间互换。

## 架构草图(直觉,不锁实现)

这是编辑器宿主层的结构改动,动的是 shell 而不是浏览器:

- **宿主布局**:#content 从单一 surface 栈变成 1-2 个 pane 容器;每个 pane = 自己的 doc-frame(iframe)或 web viewport 锚点。现有 surface 互斥逻辑(doc/viewer/newtab/web)按 pane 实例化。
- **shell 状态**:docPath/dirty/autosave/undo 全部从模块级单例改成 per-pane——**这是最大的一刀**(shell.js 里一堆模块级 let)。建议先做"pane 上下文对象"重构(纯机械,单独 PR),再上分屏。
- **web view**:bounds 按 pane 矩形算(现有 bounds() 改带 pane 参数);两格都是网页 = 两个 WebContentsView 同时 attach(现排他不变式要放宽为"每 pane 至多一个"——排他探针跟着改)。
- **持久化**:v1 不持久化分屏(重启回单栏),省一轮状态迁移。
- **fit-to-width**:每格独立算(bounds 变化已挂 refit,天然适配)。

## 单元切分(估 2-3 周,含前置重构)

- U0 **前置:shell per-pane 上下文重构**(不改行为,纯机械;e2e 全套守恒)——单独 PR
- U1 拖拽吸附 + 分屏布局 + 分隔线(e2e:拖 b.html 到右缘 → 两格各自渲染)
- U2 焦点格模型 + 快捷键路由(e2e:焦点在右格 Cmd+W 只关右格;变异自检:焦点指示必须随点击移动)
- U3 网页格(bounds per-pane + 排他不变式放宽 + 两 view 并存;e2e:左网页右文档,滚动互不干扰)
- U4 脏态/自动保存/关闭守卫 per-pane(e2e:右格脏、关左格不弹确认)
- U5 收尾:侧栏激活态变体 + 冒烟 + 文档
