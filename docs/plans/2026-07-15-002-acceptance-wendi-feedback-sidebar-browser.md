# Wendi 2026-07-15 五条反馈修复 · 验收测试 plan（U1–U5）

## 0. 这份 plan 是干什么的

原始 plan（[`2026-07-15-001`](2026-07-15-001-fix-wendi-feedback-sidebar-browser-plan.md)）定义了 Wendi 五条
反馈怎么修，已按 U1→U5 全部实现并合入 main（PR #215 = U1+U2、#217 = U3、#221 = U4+U5）。**本 plan 不写
新代码**，它是一张**验收清单**：逐条确认每个改动「确实实施了」且「行为符合当初拍板的预期」，并明确
**自动化门覆盖到哪、覆盖不到哪、宿主真交互必须补哪**。执行者可以是人（Colin/Wendi 真机试）或另一个
model（跑门 + 读断言强度）。

铁律沿用本仓 S4：**代理断言（查 class / 查「没报错」）≠ 视觉/行为验证**。下面每条都标了门「为什么强」
（能想出一种「实现全废但断言还过」的情形，它就还是弱的——这类一律点名）。

## 1. 前置与环境坑（先读，否则会把绿看成红）

- **跑自动化门**：`cd` 到仓库根，`npm run test:e2e:dot`（全量，约 10 分钟，CI 每 PR 也跑）；或定向
  `npx playwright test e2e/<spec>.spec.js -g "<用例名>"`。
- **⚠ 本机暗色模式会假红 3 条**：`align.spec.js` T1/T2 + `nonconform-basic-edit.spec.js` T5 是 computed-style
  **色值**断言，在 macOS 系统深色下取到暗态色值（如焦点框 `rgb(92,159,223)` vs 亮态 `rgb(29,111,191)`）
  会失败——**这不是本批改动的回归**，CI 亮色 xvfb 环境全绿。验收时把这 3 条排除（或临时切系统浅色再跑）。
  判据：这 3 条在**本批任何改动之前的 main**上、暗色机器上同样红。
- **权威门是 CI**：main 有 required check = `test` + `e2e`，PR 落后 main 会被拒合。本地绿只是省一次
  往返，最终以 CI 亮色环境为准。
- **宿主真交互验收放 macOS 真机**：容器无显示器、跑不了真窗口；且下面多条的「原生菜单加速器 / 真键盘」
  维度**自动化根本测不到**（见 §7），必须在装了新版 app 的 mac 上手动走一遍。

## 2. U1 · 空态「打开文件夹」CTA 挪到侧栏最底

- **验收目标**：0 个工作区根、但侧栏因有网页标签而亮起时，空态「打开一个本地文件夹…」+「打开文件夹」
  按钮应在侧栏**最底部**（收藏区 / 置顶区 / 标签页区**之下**），不再挤在收藏区里。按钮接线未断（点了真能选文件夹）。
- **已有自动化门**：`e2e/browser.spec.js` › `U1 空态「打开文件夹」CTA 在侧栏最底（…）`。
- **门为何强**：不是查「`#sb-empty` 可见」（那是弱的——它在原来的错误位置也可见）。是**坐标断言**：
  读 `#sb-empty` 的 `getBoundingClientRect().top` 与 `#sb-tabs` / `#sb-pinned` 的 top 比大小，断言
  `emptyY > tabsY && emptyY > pinnedY`——布局功能测不出、必须坐标验。且断言 `#sb-empty-open` 可见（按钮
  真在、接线是 `pickFolder` 未断）。
- **自动化覆盖不到的**：视觉美观（间距/对齐）、真的点按钮弹出系统选择框（e2e 只验按钮存在，不点原生对话框）。
- **宿主手动验收**：装新版 → 不打开任何文件夹 → ⌘T 开一个网页标签让侧栏亮起 → 看空态 CTA 是否在最底、
  在置顶/标签页区之下 → 点「打开文件夹」按钮确认真弹系统选择框、选完真加根。
- **通过标准**：坐标门绿 + 宿主目视 CTA 在最底 + 按钮真能选文件夹。

## 3. U2 · 收藏/置顶/标签页三栏折叠统一（Colin 拍板：都可折叠，置顶/标签页默认展开）

- **验收目标**：置顶区、标签页区都能像收藏区一样点栏标折叠/展开；栏标带 caret + 计数；**默认态**=收藏
  收起、置顶/标签页展开；折叠态**持久化**（重启后保持）。栏标「+」按钮的键盘操作不误触折叠。
- **已有自动化门**：
  - `e2e/browser.spec.js` › `U2 三栏折叠统一：置顶/标签页可折叠(默认展开)+计数+持久化`。
  - `e2e/browser.spec.js` › `U2b 折叠栏标键盘：栏标 keydown 折叠、+ 按钮的 keydown 不冒泡成折叠(审查 P3 回归门)`。
- **门为何强**：
  - 折叠：点 `.sb-zone-caret` 后断言 `.sb-zone-list` **`toHaveCount(0)`**（列表真不渲染，不是查 class 名），
    且 `is-open` 撤、`.sb-zone-count` 出真实计数文本（`'1'`）。
  - 持久化：`app.close()` 重启后断言 `.sb-zone-count` 仍 `'1'` 且仍 `not.toHaveClass(/is-open/)`——**用计数
    证明标签会话真恢复了**（不是空壳折叠态）。
  - U2b（审查抓出的 P3 回归门）：用 `dispatchEvent('keydown', {key:'Enter'/' '})` **确定性**派发（不靠 xvfb
    下不可靠的 OS 焦点路由），验 `head.onkeydown` 的 `e.target===head` 守卫——栏标自身键触发折叠、从「+」
    按钮冒泡上来的键**不**触发折叠。这一条正是审查发现「+ 按钮 Enter 会误折叠且吞掉按钮动作」后加的门。
- **自动化覆盖不到的**：caret 旋转动画的视觉、三栏折叠后的整体视觉协调、ui-demo 侧的对齐（ui-demo 已同步
  改，见 `ArcSidebar`——ui-demo 是 React mock，验收看 Vercel 预览）。
- **宿主手动验收**：装新版 → 有工作区 + 若干标签 → 点置顶/标签页栏标确认能折叠、caret 转、计数对 → 重启
  app 确认折叠态还在 → 键盘 Tab 到「+」按钮按 Enter 确认是「新建」不是「折叠」。
- **通过标准**：U2 + U2b 门绿 + 宿主三栏都能折叠 + 重启保持 + 「+」键盘不误折叠。

## 4. U3 · 浏览器原地导航加载期标签行 spinner（治「闪回旧页面 1-2s」）

- **验收目标**：网页标签**原地导航**到新 URL 时，加载期渲染区仍显示旧页（Chrome 语义、不摘 view），
  但**标签行出现转圈 spinner** 作加载指示；提交后 spinner 熄、新页上屏。导航时序**不改**，只补可见反馈。
- **已有自动化门**：`e2e/browser.spec.js` › `U3 导航加载反馈：慢站导航期标签行转圈(spinner 真动画,非查
  class)、旧页保留、提交后消失`。
- **门为何强**：spinner 不是查 `.is-loading` class（那是弱的——class 是 JS 直接设的、样式全废也过）。是
  **强断言 `getComputedStyle(el, '::after').animationName` 含 `ws-spin`**——真取到旋转动画的 computed 值，
  样式没生效就红（哑门自检）。且用 view 中心**像素**断言「加载期仍是旧红页」（`isRed`）、「提交后是新蓝页」
  （`b>120 && r<90`）——不是查 JS 状态，是查 view framebuffer 真色。变异自检：打掉 `setTabLoading` → 翻红（已验）。
- **审查连带修的 P2**（已合在同 PR）：渲染进程**加载中崩溃**时补 `r.loading=false`——否则新 spinner 会
  永久旋转。此路径 e2e 未覆盖（crash e2e 脆弱），靠代码审查 + 与既有 `did-fail-load` 同款写法保证。
- **自动化覆盖不到的**：spinner 的视觉观感（大小/位置/是否遮住 favicon 不适）、真实慢网站上的主观「不再
  以为卡死」体验（Wendi 的原始诉求是感知层）、崩溃期 spinner 熄灭（P2 修，无 e2e）。
- **宿主手动验收**：装新版 → 开网页标签 → 地址栏输一个**慢**站（或 devtools 限速）回车 → 确认加载期旧页
  还在 + 标签行转圈 + 提交后转圈停、新页上；重点问 Wendi「现在还觉得像卡死/闪回吗」。
- **通过标准**：U3 门绿（含动画强断言 + 像素）+ 宿主看到转圈 + Wendi 确认感知改善。

## 5. U4 · ⌘\ 切换侧栏——全焦点可用（含文档编辑 iframe 内）+ 菜单可发现

- **验收目标**：⌘\ 在**所有焦点域**都能切侧栏——主界面、**文档编辑器 iframe 内**（Wendi 报的失灵域）、
  网页标签聚焦时；且应用菜单「视图 → 切换侧栏」可见可点。单次生效、不双触发（切一次不是两次）。
- **已有自动化门**：
  - `e2e/tabs.spec.js` › `UX-U4: ⌘\ 视图菜单 toggle-sidebar → 折叠/恢复；文档编辑聚焦时仍生效`。
  - 既有 `e2e/workspace.spec.js` › `Cmd/Ctrl+\ 收起/展开侧栏` + `e2e/sidebar.spec.js` › `收起态不再有
    竖排图标轨` —— 验主层 `document` keydown fallback（用 `page.keyboard.press('Control+\\')`）。
- **门为何强 / 弱**：
  - UX-U4 验的是 **onMenu 路由契约**：`webContents.send('menu','toggle-sidebar')` → 侧栏 `is-collapsed` +
    `#sb-reopen` 可见 → 再触发恢复。这能抓到「菜单路由没接」的回归（老代码无此菜单路由 → 会漏）。
  - ⚠ **诚实标注的弱点（审查指出）**：UX-U4 用的是 `send('menu', …)` **IPC 直投**，**绕过了原生菜单
    加速器**。所以它验的是「给定一个 menu 事件，路由是否正确」，**不是**「真按 ⌘\ 时原生加速器在各焦点域
    是否真的触发」。特别是它虽然点进了 `#doc-frame` iframe 再触发，但因为投的是 IPC，**即使原生加速器
    在 iframe 里根本不触发，这条也会绿**——它宣称的「原失灵域回归门」并没有被端到端真正压住。这个维度
    **只能靠宿主真键盘验**（§7）。
  - keydown fallback 门（workspace/sidebar）用 CDP 注入按键、走 DOM keydown 这条 fallback，验的是 fallback
    真在（我执行时一度把它误删、这两条门当场翻红，是它们逼我把 fallback 加回来的——有牙）。
- **实现要点（验收时对照代码在不在）**：`main.js buildMenu` 有「视图」子菜单含 ⌘\；`shell.js onMenu` 有
  `toggle-sidebar → __sbHooks.toggleSidebar`；`sidebar.js` 主层 keydown **保留**作 fallback；`web-tabs.js`
  的 `'\\'→'toggle-sidebar'` 转发 + `browser.js onWebShortcut` 接收端**已删**（web 焦点双触发源）。
- **宿主手动验收（这条最关键，自动化替不了）**：
  1. 点进文档正文（焦点在 `#doc-frame` iframe 内）→ 真按 ⌘\ → 侧栏应切换（**这是 Wendi 原始 bug 的直接
     复现，必须真键盘、真在编辑器里按**）；再按恢复。
  2. 焦点在主界面 → ⌘\ 切换。
  3. 开网页标签、点进网页内容让 web view 聚焦 → ⌘\ 切换（验删转发后 web 焦点仍靠菜单加速器生效、且**只切
     一次不是两次**——盯有没有双触发导致「按了跟没按一样」）。
  4. 菜单栏点开「视图」→ 看到「切换侧栏 ⌘\」→ 点它也能切。
- **通过标准**：UX-U4 + 两条 fallback 门绿 + **宿主三个焦点域真键盘各单次生效（尤其编辑文档时）** + 菜单可见。

## 6. U5 · ⌘R = 刷新当前网页标签（文档标签有意 no-op）

- **验收目标**：网页标签激活时 ⌘R 真刷新当前页（与 Chrome 一致）；**文档标签激活时 ⌘R 什么都不做**
  （不重载、不丢未保存编辑）；起始页（url=null）⌘R 不炸。
- **已有自动化门**：
  - `e2e/browser.spec.js` › `U5 ⌘R 刷新网页标签：菜单 reload → server 真收到二次请求（命中计数+1…）`。
  - `e2e/browser.spec.js` › `U5 边界 ⌘R 无可刷新目标不炸`。
  - `e2e/tabs.spec.js` › `UX-U5: ⌘R 对文档标签 no-op —— menu reload 不重载 iframe（易失 DOM 标记存活…）`。
- **门为何强**：
  - web 刷新：新增 `/rl` 路由每次请求 `rlHits++`，触发 reload 后断言 `rlHits` 从 1 → **2**——**server 端
    命中计数**证明 view 真回源重载（`Cache-Control: no-store` 防吃缓存），不是查 renderer JS 状态。变异
    自检：把 `__webMenu` 的 `navReload.click()` 去掉 → `rlHits` 停在 1 → 翻红（已验）。
  - 文档 no-op：往 `#doc-frame` body 塞一个磁盘原文没有的**易失 DOM 标记**，触发 reload 后断言标记**仍在**
    ——若误重载 iframe，磁盘原文会把标记冲掉。**查存活标记，不是查「没报错」**（弱门只会查没抛异常）。
    变异自检：给 shell.js 误加一句文档 reload → 标记被冲 → 翻红（已验）。
  - 边界：无网页标签时触发 reload → 断言 `attachedCount()===0`（没误开导航）+ 壳没崩。
  - ⚠ 同 U4：这三条都用 `send('menu','reload')` IPC 直投，**绕过原生 ⌘R 加速器**——验路由，不验真键盘。
- **自动化覆盖不到的**：真按 ⌘R 的原生加速器在网页 view 聚焦时是否真触发；⌘R 落在子页面（设置/历史/收藏页）
  开着时会先关子页再刷后台页（审查记的良性交互）；起始页 url=null 的 `navReload` disabled 守卫走真键盘。
- **宿主手动验收**：
  1. 网页标签、点进网页 → 真按 ⌘R → 页面真刷新（找个带时间戳/计数的页看它变）。
  2. 打开一个**有未保存编辑**的文档标签 → 真按 ⌘R → 确认**什么都没发生、未保存的字还在**（§2 拍板的关键
     安全线，Wendi 编辑时误按不能丢东西）。
  3. 起始页标签（url=null）→ ⌘R → 不炸。
- **通过标准**：三条门绿 + 宿主网页真刷新 + **文档标签 ⌘R 真不丢未保存编辑** + 起始页不炸。

## 7. 跨切面：自动化测不到的三件事（验收必须靠宿主/CI 补）

1. **原生菜单加速器 vs IPC 直投的 gap（U4/U5 最关键）**：所有 menu-route e2e 用 `webContents.send('menu',…)`
   直投，**绕过原生加速器整层**。它们证明「事件到了、路由对」，**不证明「真按 ⌘\/⌘R 时原生加速器在
   iframe / web view 焦点下真的会触发」**。这一维度自动化在容器里根本做不了（真键盘 + 真焦点 + 原生菜单），
   **必须宿主 macOS 真机手按**（§5.4、§6.4 已列步骤）。这是整批里最容易「绿但没真验」的地方。
2. **跨平台假设未验（Windows/Linux）**：U4/U5 依赖「原生菜单加速器在焦点落在子 WebContentsView 上时也
   触发」，仅 macOS 由 ⌘W（不转发却能浏览时关标签）间接实证。U4 删了跨平台的 `before-input` 转发后，web
   焦点下无 fallback——若某平台加速器够不到聚焦的子 view，⌘\/⌘R 浏览时会失灵。已记 `docs/features/browser.md`
   欠账，**真机闭环放到 Windows 打包冒烟一并做**（本 feature Windows 整体本就未验）。
3. **暗色模式 3 条假红非回归**：见 §1，验收读结果时先排除，别当成本批引入的 bug。
