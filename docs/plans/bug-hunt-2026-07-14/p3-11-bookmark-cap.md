# [P3-11] 收藏无条数上限(历史有 500 cap)

**问题(观察项)**:导入 8000 条书签全收,`browser-bookmarks.json` ~1.5MB,且每次变更全量推给 renderer。
不崩、功能正常——是「没有防线」而不是「坏了」。

**修法(最小防线,别过度设计)**:
1. **不设硬上限**(收藏是用户数据,静默丢弃比膨胀更糟)。
2. 只修放大器:变更推送从「全量 state」改「全量但防抖合并」(200ms 内多次变更推一次)——
   先 grep 现状,若已有防抖则此项关闭;
3. 导入超大文件(>5000 条)时 toast 提示一句「已导入 N 条」(用户至少知道发生了什么)。

以上任一项若实测已存在,就地关闭该项,别造重复轮子。总体这是三档里最低优先级的计划,排最后做。

**门**:单测:防抖合并(计推送次数);e2e 不需要。

**spec 记账**:browser-feature-spec.md §收藏(容量语义:无上限+防抖推送)。
