**本地优先的文档与发布平台**

Wordspace 是一款以浏览器为形态的文档工具。每一份文档都是一个独立的 HTML 文件，默认保存在使用者自己的设备上，也可一键部署到其掌控的服务器。文档由人与 AI 共同创作，既供人阅读，也供软件 Agent 调用。

## 1 概述

Wordspace 的出发点是一个判断：今天团队赖以协作的文档工具，把内容、格式与数据一并绑定在服务商自己的系统里。使用者写下的东西既离不开那个平台，也无法以通用的形式留在自己手中。一旦停止付费或平台关停，内容的可用性便不再由使用者决定。

Wordspace 的做法相反。它以 HTML 作为一切文档的统一格式。每一份文档就是一个可以独立打开的 HTML 文件，默认保存在使用者本地，归使用者所有；需要分享时，使用者一键即可把它部署到自己掌控的服务器，变成内网页面或对外网站。

由于文档本身就是 HTML，而浏览器原本就是打开 HTML 的工具，Wordspace 把这两件事合在一起：它既是创作与管理文档的地方，也是一款能够打开任意网页的浏览器。创作也不限于人。文档既可以由人在编辑器中写就，也可以交给 AI 生成；做好的文档既供人阅读，也供软件 Agent 读取与调用。

## 2 缘起：现有工具的局限

Notion 把文档、数据库与协作做得相当完整，代价是所有内容都存放在它自己的服务器上。使用者要取回内容，需要专门导出；离开 Notion 的环境，导出的内容也难以原样使用。其格式能力同样有限，止步于较为基础的样式，复杂的版面无法实现。

把视野放宽，问题不止于某一个工具。个人笔记、团队内网、对外官网，在今天分属不同的软件、不同的格式、不同的发布流程。同一段内容若要在这三种场合出现，往往要在三套工具里各做一遍，彼此之间无法沿用。

Wordspace 认为，这三者在本质上是同一份内容面向不同人群的不同可见范围。要把它们统一，前提是先统一格式。HTML 正是那个所有设备都能原样打开、又足以承载复杂版面的通用格式。

## 3 核心理念

### 以 HTML 为统一格式

HTML 是描述网页内容与版式最通用的格式，几乎所有设备与浏览器都能将其原样打开。以它作为文档格式，文档的表达力便不再受限于某个编辑器预设的几种样式，而可以达到一个真实网页所能呈现的程度。同样重要的是，HTML 文件自包含、可读、不依赖特定厂商，多年以后用任何一款浏览器打开，它依然是它本来的样子。

### 本地优先，数据归使用者所有

文档的正本保存在使用者本地，以仓库的形式组织，即设备上的一组文件与文件夹，可以离线打开、随时备份、整体迁移。当需要协作或对外发布时，文档被部署到使用者或其公司自己掌控的服务器，而非 Wordspace 的服务器。数据是否真正归使用者所有，取决于它存放在何处；Wordspace 在结构上始终把正本留在使用者一侧。这样，即便有朝一日不再使用 Wordspace，内容仍完整地留在使用者手中。

### 创作与发布是同一件事

在多数工具里，撰写文档与把它发布成网页是两个阶段、两套机制。在 Wordspace 里，文档既然已经是 HTML，发布便只是把同一个文件放到一个可被访问的地址上。于是从一份私人草稿，到邀请几位同事协作，到全公司可见的内网，再到面向公众的官网，自始至终都是同一份文件，改变的只是它的可见范围。

### 人与 Agent 共同的工作对象

文档是 HTML，这种格式同样便于软件稳定地解析与生成。Wordspace 因此从设计之初就把 Agent 视作正式使用者：Agent 既可以调用 Wordspace 来生成文档，也可以读取已有文档，作为其工作的输入。写给人看的内容与供程序处理的内容，在这里不必分作两份。

## 4 产品形态

### 作为浏览器

Wordspace 在形态上是一款浏览器。使用者下载它、把它当作日常浏览器使用，与更换一款浏览器并无不同。它渲染来自互联网的网页与渲染本地的文档，用的是同一套机制，因为两者都是 HTML。文档能力内建于浏览器之中，而非另一个需要单独打开的应用。这一选择也关乎分发：让人接受一款新的浏览器，比让人再安装一个笔记软件，门槛更低。

### 文档即 HTML 文件

每一份文档对应一个 HTML 文件。它可以朴素得像一页纯文字笔记，也可以复杂得像一个带版式、配图与样式的完整网页。两者是同一种东西的两端，使用同一套存储与发布方式。文件之间通过普通的超链接相互引用，这本就是 HTML 原生的能力。

### 存储：本地仓库与可部署的云端

文档以仓库的形式保存在本地，即使用者设备上的一组文件与文件夹，这是文档的正本，可离线访问。当文档需要被他人看到时，使用者一键将其部署到一处云端地址。这处云端由使用者或其公司掌控，可以自行托管，也可以整体迁移，不锁定在 Wordspace 的服务之上。本地正本与云端副本之间保持同步，其具体机制见第九节。

### 创作与编辑

文档有两种地位相当的创作方式。其一，在所见即所得的块编辑器中，像使用 Notion 那样逐块输入文字、标题、图片与列表；其二，用自然语言向 AI 描述，由 AI 直接生成整篇文档的 HTML。两者可以来回衔接：AI 生成的文档能在编辑器里继续修改，编辑器里写就的内容也能交给 AI 重做。

修改文档时，人手操作与 AI 操作有明确分工。日常的、局部的改动由人直接完成，包括修改文字、调整字体字号等基本格式、把段落或板块上下移动。涉及版面与视觉的较大改动，例如把一段内容做成带背景的横幅、为整页重新排版、更换整体风格，则交由 AI 完成。

这一分工使 Wordspace 不必去构建一个完整的可视化网页编辑器，也就是像 Webflow 那样直接拖拽、改写任意 HTML 元素的工具。这类编辑器在工程上极重，而 Wordspace 把对应的能力交给了 AI：人手只负责轻量改动，简单可靠；复杂的设计由 AI 承担。

为使人手的轻量编辑能够作用在一份由 AI 自由生成的 HTML 上，生成环节需遵守一定的结构约定：板块须是可整体移动的单元，文字须是可就地编辑的区域。AI 在视觉上拥有充分自由，但其输出需保持一种人手工具能够识别并操作的结构。

### 模板

Wordspace 内置模板库。公司的手册、标书、演示文稿等常用文档的版式，可以保存为模板；此后无论是人还是 Agent，都能调用模板，快速生成一份带有公司既有样式的文档。模板有两类来源：公司内部私有的模板库，以及公开共享的模板池。后者允许跨组织复用，Agent 亦可从中取用。

### 发布与权限

发布是把一份文档放到一个可访问地址的动作。同一份文档拥有一条连续的可见范围，使用者借此决定谁能看到它。

| 可见范围 | 说明 |
| --- | --- |
| 仅自己 | 文档仅存于本地或仅本人可见，用于起草阶段。 |
| 受邀协作者 | 邀请指定成员共同编辑或查看同一份文档。 |
| 全公司内网 | 部署至公司内网，组织内成员均可访问。 |
| 互联网公开 | 生成公开地址，任何人皆可打开，等同于一处对外站点。 |

由于内网页面与对外官网在 Wordspace 里都只是被部署、并被设定了可见范围的 HTML 文档，公司的内部知识库与对外站点可以由同一套工具、同一批文件来维护，无须再分属两处。

### 实时协作

受邀的协作者可以同时编辑同一份文档，改动实时可见。这一能力与本地优先的存储之间需要协调：文档的正本在本地，而多人实时编辑要求一处共享且即时一致的状态。两者如何并存，是 Wordspace 的关键设计问题之一，详见第九节。

### 面向 Agent

Wordspace 向 Agent 开放接口，使其成为与人对等的使用者。作为生产者，外部 Agent 可以调用 Wordspace 创建并生成文档，例如自动产出一份报告或一个页面。作为读者，Wordspace 产出的文档可被 Agent 读取，作为其后续处理的输入。文档统一为 HTML，使同一份内容既适合人阅读、又便于程序解析，这是 Agent 得以自然接入的基础。

## 5 一篇文档的生命周期

一份文档从产生到对外，可以把上述各项能力串成一条完整的路径。它从创建开始，或在块编辑器中写就，或由 AI 依据一句描述生成，或基于一份现成模板套出。创建之后进入修改，人调整文字与顺序，AI 负责版面与设计上的较大改动。整个过程中，文档始终以 HTML 文件的形式保存在本地仓库，归使用者所有。

当需要他人看到时，使用者一键发布，并选定可见范围，从仅自己，到受邀协作者，到全公司，直至面向公众。此后，同事、协作者、公众或 Agent 在各自被授予的范围内访问这份文档。若需向公司之外提交，文档可导出为 PDF、Word 或演示文稿，以满足平台之外的接收方。同一份文件，贯穿起草、协作、发布与交付的全过程。

## 6 与现有工具的关系

Wordspace 的目标，是成为团队办公文档创作与协作的主要场所。就文档而言，它对标并意在替代 Notion；在公司内部，它也承担原先由 Word 与演示文稿软件完成的起草工作。

替代有其边界。公司之外的世界仍在使用 .docx、.pptx 与各类在线文档。只要还需要把文件交付给客户、合作方或外部机构，Wordspace 就需要能导出为对方所用的格式。因此更准确的说法是：公司内部的起草与协作集中在 Wordspace 中完成，对外则导出为接收方需要的格式，而不追求让外部世界也改用它。

与 Google Drive 一类存储与协作服务，Wordspace 在相当长一段时间内仍是共存关系，而非立即取代。至于本地存储，它借鉴了 Obsidian 的思路，即内容以普通文件的形式存在本地、归使用者所有；取用的是其中“本地与归属”这一层，而非 Obsidian 的双向链接与知识图谱。

## 7 范围与取舍

为使第一阶段的目标清晰，有必要写明当前不做什么。

带结构的数据内容暂不涉及。数据库、表格、OKR、项目看板这类依赖状态、筛选与多视图的内容，与文档是不同的产物；Wordspace 当前只处理文档。它们是否以及如何纳入，留待文档部分成熟之后再议。

Wordspace 也不在近期构建完整的可视化网页编辑器。任意 HTML 元素的可视化改写已交由 AI 承担（见第四节），人手工具只覆盖轻量编辑。这既是为了控制工程量，也是为了让产品的重心落在创作与发布的流程上，而非编辑器本身。

对 Word 与演示文稿软件的全面取代不是近期目标。近期只要求在公司内部承担起草与协作，对外通过导出与既有格式兼容。

## 8 第一阶段目标

第一阶段的目标具体而有限：让 Wordspace 先替代团队当前在 Notion 中的文档使用。第一批文档是团队日常的战略文档、业务笔记与各类手册，也就是不依赖表格与数据库的纯文档内容。我们以自身团队（Tenth Global）作为第一批使用者，在真实使用中检验产品，再向外推广。

衡量第一阶段是否达成，要看几件事能否同时成立：团队的日常文档能在 Wordspace 中撰写，并支持多人实时协作；这些文档能部署到团队自己掌控的服务器，在公司内部访问；以及“把一份页面一键发布为真实网页”这一典型场景能够完整实现，招聘页便是其中的代表。

## 9 关键设计与技术挑战

有几个问题决定了 Wordspace 能否成立，在此一并写明，而不当作已经解决。

#### 本地正本与实时协作的协调

*文档正本在本地，而多人实时编辑要求一处即时一致的共享状态。*

如何在保证数据归属的前提下实现流畅的实时协作，包括编辑冲突的合并、离线之后的重新同步，是首要的设计问题。本地与云端两份内容必须既保持一致，又不剥夺使用者对正本的掌控。

#### AI 生成结果的结构约束

*AI 在视觉上自由，但其产出需保持人手工具可识别、可操作的结构。*

若 AI 生成的 HTML 毫无章法，就地编辑与板块移动便无从着手。如何在不牺牲设计自由的前提下约束生成结构，需要一套专门的规则与约定贯穿于生成环节。

#### 导出的保真度

*把一份 HTML 文档导出为 .docx、.pptx 或 PDF，涉及不同格式之间版式的转换。*

导出结果能否达到可直接对外提交的质量，直接影响 Wordspace 在公司内部能在多大程度上替代既有工具。这一环若不过关，内部统一便会在对外交付处断开。

#### 浏览器一层的实现深度

*第一阶段是构建一个真正的浏览器内核，还是以一个能渲染并管理 HTML 的应用外壳起步。*

两条路线在工程量与分发策略上差别很大。前者更贴近“取代浏览器”的设想，后者更快进入可用状态。何时走向哪一条，需在第一阶段开始之前权衡定夺。
